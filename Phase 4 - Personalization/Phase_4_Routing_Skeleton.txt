------------------------------------------------
Phase 4 CI вЂ” Governance & Safety Layer
------------------------------------------------

Scope:
- Applies ONLY to Phase 4 (Personalization)
- Enforces invariants without modifying runtime behavior

Execution Context:
- CI-only
- NEVER part of runtime execution
- NEVER invoked by Phase 3 Orchestrator
- NEVER invoked by Phase 6 API

------------------------------------------------
CI Categories (Phase 4)
------------------------------------------------

1. Determinism & Regression
   Files:
   - ci/determinism_checks.py
   - test_deterministic_core_invariants.py
   - test_fixture_determinism_regression.py
   - fixtures/fixture_deterministic_input.json

   Purpose:
   - Ensure Phase 4 does not break deterministic guarantees
   - Detect accidental behavioral drift via golden fixtures
   - No judgement of personalization quality

------------------------------------------------

2. Safety & Guardrails
   Files:
   - ci/safety_checks.py
   - test_safe_adjustment_bounds.py
   - test_personalized_fixture_bounds.py
   - fixtures/fixture_personalized_input.json

   Purpose:
   - Ensure safe adjustment layer exists and is not bypassed
   - Ensure selected elements are not semantically mutated
   - Enforce bounded, non-destructive personalization

------------------------------------------------

3. Explainability & Contract Integrity
   Files:
   - ci/explainability_checks.py
   - test_personalization_decision_schema.py
   - test_event_logging_contract.py
   - test_personalized_fixture_bounds.py

   Purpose:
   - Enforce explainability chain:
     decision_source в†’ model_outputs в†’ applied_adjustments в†’ provenance
   - Validate presence and parseability of contract docs and schemas
   - Ensure event logging schema exists

------------------------------------------------
Forbidden CI Behavior (Hard Rules)
------------------------------------------------

- Phase 4 CI MUST NOT:
  - judge model quality or ranking performance
  - enforce language or narrative quality
  - mutate Phase 4 runtime behavior
  - gate runtime execution directly
  - bypass Phase 6 governance
  
------------------------------------------------
Phase 4 CI вЂ” Tests vs Checks Separation
------------------------------------------------

Phase 4 CI is explicitly divided into two layers:

1. Checks (Governance / Contract Layer)
   Location:
   - Phase_4_Personalization/ci/*_checks.py

   Purpose:
   - Enforce non-negotiable invariants
   - Validate contracts, schemas, and guardrail existence
   - Prevent silent semantic drift

   Properties:
   - CI-only
   - No fixture execution
   - No runtime mutation
   - Authoritative for governance

------------------------------------------------

2. Tests (Execution / Regression Layer)
   Location:
   - Phase_4_Personalization/tests/
   - Includes fixture-based and runtime-executing tests

   Purpose:
   - Validate deterministic behavior
   - Detect regression via golden fixtures
   - Assert bounded and explainable personalization behavior

   Properties:
   - Executes Phase 4 runtime
   - Uses fixtures and golden hashes
   - Does not judge model or narrative quality

------------------------------------------------

Runner Exception:
- run_all_personalization_checks.py
  - Belongs to the test layer
  - Acts as a test orchestrator only
  - Never considered a governance check

------------------------------------------------
Relationship to Other Phases
------------------------------------------------

- Phase 4 CI constrains Phase 4 runtime only
- Phase 4 CI has NO authority over Phase 1вЂ“3 semantics
- Phase 4 CI emits no signals to Phase 5 learning
- Phase 4 CI is advisory to developers, authoritative to CI
