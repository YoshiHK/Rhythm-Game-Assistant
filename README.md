## Rhythm Game Assistant (RGA)

Rhythm Game Assistant is a system-driven rhythm game companion that recommends songs and provides gameplay tips generated by deterministic analysis pipelines.
This repository contains the **source code, API wiring, and system documentation** required to:
- Generate gameplay tips offline via batch ingestion
- Serve personalized recommendations and tips to a frontend (Softr)
- Preserve strict phase boundaries and semantic immutability

**Important:** This repository does NOT store chart files, generated databases, or player data.
Charts are ingested offline from a OneDrive directory via a scheduled job.

### 1. High-level architecture

[ OneDrive Charts ]
 ↓ (03:00 Scheduled Job)
[ Phase 1–3: Tip Generation + UMI ]
 ↓
[ Tips Database (generated outputs only) ]
 ↓
[ Thin Backend API (this repo) ]
 ↓
[ Softr App (UI / consumption surface) ]

**Core rules:**
- Players never upload charts
- Players never trigger analysis
- Tips meaning is immutable once generated
- Personalization and localization are presentation-only

### 4. Backend API (Softr integration)

This repository exposes a **thin HTTP API** intended for UI consumption only.

#### Example endpoint (current v1)
POST /api/v1/recommend

**Responsibilities:**
- Authenticate Softr via Bearer token
- Accept player performance signals
- Return read-only recommendation results and rationale
- Return locale and provenance_id for downstream presentation and learning linkage
- Provide client event taxonomy for Phase 5 feedback aggregation

**Non-responsibilities:**
- Chart ingestion
- Gameplay analysis
- Tips generation
- Model training
- Reading raw Song Database files (e.g. Excel) at runtime

**See also:** api/routes/recommend.py

### 5. Authentication model

Softr authenticates using a **service-to-service API key**.

**Header format:** Authorization: Bearer <API_KEY>

**Details:**
- API key is validated by the backend
- This key authenticates Softr as a **client**, not individual players
- Follows Phase 6 (Platform Hardening) principles

### 6. Tip generation workflow (offline)

Tip generation is fully automated and runs **outside** this API:
- Charts live in a OneDrive folder
- A scheduled job runs UMI daily
- Tips and summaries are generated deterministically
- Only generated outputs are persisted

The backend API **never** runs Phase 1–3 logic.

### 9.5 CI and system invariants

This repository uses a **Continuous Invariants (CI)** system to ensure that completed phases remain **semantically immutable** as the platform evolves.

**CI enforces:**
- Phase boundaries (no upstream semantic access)
- Deterministic behavior guarantees
- Personalization and localization safety
- Versioned, machine-consumed log-level contracts
- Operational observability and alert gating (Phase 6 wiring)

**CI does not:**
- Judge gameplay quality
- Retrain models
- Generate content

All CI rules, contracts, and observability hooks are documented in CI/README.md.

### 10. Key governance rules

#### Semantic immutability
Completed phases (1–6) MUST NOT be retroactively modified.
Any behavioral change must be introduced via:
- New phases, or
- Control-plane wiring only

#### Phase boundary rule
**Phase 3 provides signals. Phase 4 makes presentation decisions.**
No exceptions.

#### Allowed extensions
- New games (via adapters + config)
- New run modes (via orchestrator extension)
- Localization wiring (Phase 4.5)
- Personalization models (Phase 4)
- Recommendation orchestration (Phase 7)
- Observability, metrics, CI enforcement

#### Prohibited anti-patterns
- ❌ Semantic fixes inside orchestration
- ❌ Game-specific branching in core orchestrator
- ❌ Silent fallback or silent retries
- ❌ QA metrics used as execution gates
- ❌ Presentation metadata influencing gameplay logic
- ❌ "Just for one game" exceptions

For full details, see doc/LIMITATIONS.md.

## Song Catalog Access Policy

The public API **never reads the raw Song Database directly** (e.g., Excel files).

All song metadata is first:
1. Ingested and validated by **Phase 3 (UMI)**
2. Canonicalized into **versioned, immutable song catalog artefacts**

The API consumes these artefacts exclusively through a **read-only SongCatalog interface**.

### Guarantees
- Determinism and auditability
- Clear phase boundaries
- No semantic drift
- Safe operation under Phase 6 hardening

Raw Song Databases are treated as **reference inputs only**, never runtime data sources.
