Phase 6 — Platform Hardening & Runtime Routing
=============================================

Purpose:
- Act as the ONLY runtime gatekeeper.
- Own authentication, guards, routing, failure isolation, observability.
- Provide platform-level coordination for:
  - song recommendations (mode="songs")
  - game recommendations (mode="games", Phase 7)

Non-Negotiable Rules:
- No UI / client may invoke Phase 1–5 or Phase 7 directly.
- No semantic logic lives in Phase 6.
- Phase 6 performs coordination, normalization, and policy enforcement only.

------------------------------------------------
Runtime Routing Overview
------------------------------------------------

ALL runtime traffic:
UI / Client / Softr
→ Phase 6 (API / Auth / Guards)
→ phase6_router.py

------------------------------------------------
Routing Policy (High Level)
------------------------------------------------

routing_policy.py evaluates request.mode:

- mode = "songs"
  → Song Recommendation Flow (Phase 6 + Phases 1–4.5)

- mode = "games"
  → Game Recommendation Flow (Phase 7)

Any other mode:
  → STOP (explicit, explainable)

------------------------------------------------
Song Recommendation Flow (mode="songs")
------------------------------------------------

1. Phase 6 receives request
   - Auth, security, abuse, compliance guards applied
   - Request normalized into routing_context

2. song_recommendation/request_normalizer.py
   - Validate canonical submission schema
   - No Proseka-specific assumptions
   - Difficulty tiers and completion labels remain opaque strings

3. song_recommendation/game_capability_resolver.py
   - Resolve game_id → game capability config
   - Provides:
     - difficulty tier ordering
     - completion ladder ordering
   - Phase 6 does NOT interpret semantics beyond ordering

4. song_recommendation/song_rec_coordinator.py
   - Coordinates deterministic recommendation generation
   - Delegates semantic computation to downstream phases
   - Assembles 3-in-a-group recommendation intent:
     - AP/FC/Clear equivalents via ladder positions

5. Downstream execution (per system spec):
   Phase 3 Orchestrator
   → Phase 1 → Phase 2
   → Phase 4 Personalization
   → Phase 4.5 Localization

6. song_recommendation/persistence_policy.py
   - action = "refresh" → no persistence
   - action = "save" → persist recommendation set
   - Apply rotation policy (e.g. max N, keep bookmarked)

7. song_recommendation/response_shaper.py
   - Assemble final API response
   - Attach diagnostics and provenance metadata

------------------------------------------------
Game Recommendation Flow (mode="games")
------------------------------------------------

Client
→ Phase 6
→ Phase 7 Router
→ Player Capability Modeling
→ Game Capability Modeling
→ Matching & Ranking
→ Explanation
→ Feedback / Observability (async)
→ Response

------------------------------------------------
Explicit Non-Responsibilities
------------------------------------------------

Phase 6 MUST NOT:
- Contain gameplay analysis logic
- Contain chart-level semantics
- Hardcode difficulty names or completion labels
- Perform localization logic
- Run CI or learning pipelines

------------------------------------------------
Design Notes
------------------------------------------------

- Multi-game support is achieved via game capability resolution,
  not by branching logic in UI or algorithms.
- Proseka is treated as one game profile, not a system baseline.
- Phase 6 remains a coordination and policy layer only.

End of Phase 6 Routing Skeleton
